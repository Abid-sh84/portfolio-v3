> **In short:** Modern search engines no longer match keywords — they convert your content and user queries into high-dimensional vectors using transformer models (BERT, Google MaLLM), then find the closest vectors in under 200ms using HNSW indexing. Ranking is a multi-signal formula combining semantic similarity, PageRank, freshness, and structured data. To rank in 2026, write for human intent — semantics handle the rest.

## From Keywords to Meaning — The Big Shift

For decades, search engines matched *exact words*. Type "best javascript framework" and they'd hunt for pages containing those three words, completely ignoring context.

## Part 1: Words to Numbers — Vector Embeddings

An **embedding** is a numerical representation of meaning. Similar meanings produce similar numbers — that's the entire premise.

<EmbeddingViz />

**How are embeddings created?** Search engines use **Transformer models** (BERT, BGE, Google's MaLLM) trained on billions of documents. The pipeline inside a transformer:

<Pipeline steps="Input text — 'The quick brown fox jumps over the lazy dog' | Tokenizer: Split into subword units | Embedding Layer: Each token mapped to a 768D vector | 12 Transformer Layers: Contextual self-attention | Output: Dense vector capturing full sentence meaning" />

Each layer adds more semantic understanding — by layer 12, the model knows the subject, the action, and the overall intent of the sentence.

---

## Part 2: The Indexing Pipeline

Here's how modern AI indexing works end-to-end, from your website to the search index:

<Pipeline steps="Stage 1 — Discovery: robots.txt + sitemap.xml, Googlebot crawls all pages | Stage 2 — Preprocessing: Extract main text, remove boilerplate (nav, footer, ads), split into 512-token chunks | Stage 3 — Semantic Encoding: Each chunk goes through transformer model, producing a 384 to 1024D vector | Stage 4 — Storage: Vectors stored in HNSW or IVF data structures enabling sublinear search" />

<Callout type="key">
  Google processes **8.5 billion pages**. At ~$0.0001 per embedding, inference costs run into millions per indexing pass — which is why they invest heavily in model quantization and distillation.
</Callout>

HNSW (Hierarchical Navigable Small World) and IVF (Inverted File Index) structures enable **O(log n) nearest-neighbor search** across billions of vectors — finding similar content in milliseconds.

## Part 3: Query Time — What Happens in Under 200ms

<Pipeline steps="You type: 'best javascript framework for web apps' | Same transformer embeds your query into a vector | Cosine similarity search across the entire index | Top 1000 candidates retrieved | Re-ranker applies multi-signal scoring | Top 10 results returned" />

**Cosine similarity** measures how aligned two vectors are (0 = opposite, 1 = identical). The key insight: the query goes through the *same* embedding model as the indexed content, ensuring alignment.

## Part 7: Build Semantic Search Yourself

```js
import { Ollama } from 'ollama';
import { HNSWLib } from 'langchain/vectorstores/hnswlib';

const model = new Ollama({ model: 'nomic-embed-text' });

// Index your blog posts
const vectorStore = await HNSWLib.fromDocuments(
  blogPosts.map(post => ({ content: post.content })),
  { embeddings: (text) => model.embed(text) }
);

// Semantic search — no keywords needed
const results = await vectorStore.similaritySearch(
  "How do I optimize Next.js performance?",
  5 // top 5 results
);
```

## Frequently Asked Questions

### What is AI indexing and how is it different from traditional indexing?

AI indexing converts web page content into high-dimensional numerical vectors using transformer models. Traditional indexing matched exact keywords using algorithms like BM25. AI indexing understands meaning and intent — so searching "best JS framework" can surface a page that says "top React alternatives" even with zero keyword overlap.

### How do vector embeddings work in search engines?

A transformer model (like BERT) reads your text and outputs a vector — an array of 384 to 1024 floating-point numbers that encodes the semantic meaning. Pages with similar meanings produce vectors that are close together in that high-dimensional space. At query time, your search phrase gets the same treatment and the nearest page vectors are retrieved.

### What is HNSW and why does it matter for search speed?

HNSW (Hierarchical Navigable Small World) is a graph-based data structure for approximate nearest-neighbor search. It allows Google to find the most semantically similar documents across billions of vectors in under 200ms — compared to O(n) brute-force which would be impossibly slow at that scale.

### What is RAG and how does it power tools like ChatGPT search?

RAG (Retrieval Augmented Generation) combines vector search with a language model. When you ask ChatGPT a web question, it: (1) embeds your query, (2) retrieves the most relevant documents from its index, (3) feeds those documents as context to the LLM, (4) generates an answer grounded in real retrieved content. This is why AI Overviews in Google pull from structured, well-cited content.

### How can I optimize my content for AI-powered search engines?

Focus on: writing with clear intent and semantic depth (not keyword density), adding structured data (JSON-LD) so AI engines can extract entity information, structuring FAQ sections which AI Overviews pull from heavily, citing sources with outbound links (signals trustworthiness), and ensuring your content directly answers specific questions — the "answer blurb" format works best.
